## Security Overview

StegoChat implements a **defense-in-depth** security architecture where each layer provides independent protection. Even if one layer is bypassed, others remain secure.

### Security Principles

1. **Zero-Knowledge Architecture** - Server never has access to plaintext messages
2. **End-to-End Encryption** - Only intended recipients can read messages
3. **Perfect Forward Secrecy** - Compromise of past keys doesn't expose previous messages
4. **Client-Side Key Generation** - Keys generated and stored on user's device
5. **Layered Defense** - Multiple encryption layers for redundancy

## 4-Layer Security Model

### Layer 1: Message Steganography (LSB Encoding)

**Algorithms:** Least Significant Bit (LSB) in 16-bit PCM WAV

**Purpose:** Obscure message presence from network observers

**Process:**
```
Message "HELLO" (5 bytes × 8 bits = 40 bits)
├─ Convert to binary: 01001000 01100101 01101100 01101100 01101111
├─ Generate 5-second white noise carrier (44,100 Hz = 220,500 samples)
├─ For each bit in message:
│  ├─ Read audio sample (16-bit signed integer)
│  ├─ Clear LSB: sample = sample & 0xFFFE
│  ├─ Set to message bit: sample = sample | (messageBit)
│  └─ Write back to audio buffer
├─ Append 16-bit null terminator (0000000000000000)
└─ Encode to Base64 and transmit

Result: Network observer sees only WAV audio data, no visible text
```

**Security Level:** Medium (steganography, not encryption)
**Rationale:** Protects against casual observation; combined with encryption layers for full security

### Layer 2: RSA-2048 Key Exchange (Session PIN Distribution)

**Algorithms:** RSA-OAEP with SHA-256

**Purpose:** Establish shared session key without server knowledge

**Process:**

```
User A Sign-Up Phase:
├─ Browser generates RSA-2048 key pair (client-side only)
├─ Public key: 294 bytes (2048 bits), sent to server
├─ Private key: 1704 bytes (2048 bits), LOCALLY ENCRYPTED with AES-256(password)
└─ Encrypted private key stored in browser localStorage

Chat Initiation Phase:
├─ Server generates 6-digit PIN: 123456
├─ PIN encrypted with User A's RSA public key → "pin1"
├─ PIN encrypted with User B's RSA public key → "pin2"
├─ Server stores encrypted pins in Session document
├─ Both pins sent to respective users over Socket.IO
│
├─ User A:
│  ├─ Receives "pin1" (RSA-OAEP encrypted with A's public key)
│  ├─ Decrypts using A's private key → "123456"
│  └─ Stores in memory for messaging
│
└─ User B:
   ├─ Receives "pin2" (RSA-OAEP encrypted with B's public key)
   ├─ Decrypts using B's private key → "123456"
   └─ Stores in memory for messaging

Result: Both users hold plaintext PIN = "123456", server never knows it
```

**Key Properties:**
- **Key Size:** 2048 bits (current standard)
- **Public Exponent:** 65537 (standard)
- **Padding:** OAEP with SHA-256
- **Key Storage:** Private keys encrypted with user password

**Security Level:** Very High (mathematical difficulty ~112 bits effective strength)
**Rationale:** Protects session establishment; prevents man-in-the-middle attacks

### Layer 3: AES-256 Symmetric Encryption

**Algorithms:** AES-256-CBC via CryptoJS

**Purpose:** Encrypt all message content with shared session key

**Process:**

```
Message "Hello World" → AES-256 Encryption
├─ Key: Session PIN ("123456") 
├─ IV: Random, generated by CryptoJS
├─ Plaintext: "Hello World"
├─ AES-256-CBC encrypt → Ciphertext: "U2FsdGVkX1+gI1Yd8Ke0UJk..."
│
├─ Convert ciphertext to binary string
├─ Embed binary into audio via LSB (Layer 1)
│
└─ Network transmission: Base64(WAV audio)

Receive Side:
├─ Extract Base64 audio
├─ Decode LSB → Binary string
├─ Binary → AES-256-CBC ciphertext
├─ Decrypt using Session PIN → "Hello World"
└─ Display to user
```

**Key Properties:**
- **Key Size:** 256 bits
- **Key Derivation:** PBKDF2 (CryptoJS default)
- **Block Size:** 128 bits (16 bytes)
- **Mode:** CBC (Cipher Block Chaining)
- **Padding:** PKCS#7

**Security Level:** Very High (256-bit key = 2^256 possible keys)
**Rationale:** Industry-standard encryption; resistant to known cryptanalysis

### Layer 4: Server-Side Security

**Mechanisms:** JWT, bcrypt, Rate Limiting, Input Validation

#### 4a: JWT Authentication
```
Login Flow:
├─ User submits credentials
├─ Server validates password against bcrypt hash
├─ If valid, generate JWT token:
│  {
│    "id": "507f1f77bcf86cd799439011",
│    "username": "alice",
│    "isAdmin": false,
│    iat: 1712675200,
│    exp: 1713280000  // 7 days
│  }
├─ Client stores token in memory (not localStorage for security)
└─ Token included in all API requests: Authorization: Bearer <token>

Server Validation:
├─ Verify JWT signature matches JWT_SECRET
├─ Check token hasn't expired
├─ Extract user ID from payload
└─ Execute requested operation as verified user
```

**Security:** Prevents unauthorized API access; stateless authentication

#### 4b: Password Hashing
```
Signup:
├─ User password: "MySecurePass123"
├─ Generate random salt (10 rounds)
├─ Hash: bcrypt("MySecurePass123", salt) 
│       → $2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcg7b3XeKeUxWdeS86.AVtXeK3m
├─ Store hash in database (NEVER plaintext)
└─ Password never sent to server via plaintext after initial signup

Login:
├─ User submits password
├─ bcrypt.compareSync(password, storedHash)
├─ Hash in-memory, compare securely
└─ No plaintext password ever logged or stored
```

**Security:** Protects against database breach; salted hashing resists rainbow tables

#### 4c: Rate Limiting
```
API Routes:
├─ 300 requests per 15 minutes per IP
├─ Exceeding limit returns HTTP 429 Too Many Requests
└─ Prevents brute-force attacks, DDoS

Auth Routes (/api/login, /api/signup):
├─ 10 requests per 15 minutes per IP  
├─ Stricter limit on sensitive endpoints
└─ Slows credential-stuffing attacks
```

**Security:** Prevents automated attacks; protects infrastructure

#### 4d: Input Validation
```
Example - Signup validation:
├─ Email: isEmail() - validates email format
├─ Username: trim(), length 3-30, escape() - prevents XSS
├─ Password: no special validation (let user choose)
├─ OTP: must be 6 digits, matches stored hash

Regex Escaping (prevents ReDoS):
├─ User input: "alice.*bob"
├─ Escaped: "alice\.\*bob"
├─ Regex: new RegExp(`^${escapedInput}$`, 'i')
└─ Prevents catastrophic backtracking
```

**Security:** Prevents injection attacks (XSS, NoSQL injection, ReDoS)

## Threat Model

### Threat Actors

1. **Passive Network Observer**
   - Sees all network traffic
   - Cannot access client encryption keys
   - Mitigation: LSB steganography + AES-256

2. **Active Man-in-the-Middle (MITM)**
   - Intercepts traffic between client and server
   - Can modify traffic
   - Mitigation: HTTPS + RSA key exchange

3. **Compromised Server**
   - Attacker gains database access
   - Mitigation: No plaintext messages stored; passwords bcrypt hashed

4. **Malicious Admin**
   - Access to server and database
   - Mitigation: Cannot decrypt messages; private keys on client devices

5. **Malware on User Device**
   - Accesses local keys and memory
   - Cannot be prevented by server-side security
   - Mitigation: User responsibility; warn on suspicious activity

### Threat Scenarios & Mitigations

#### Scenario 1: Network Traffic Interception

**Threat:** Attacker captures message between User A and Server

**Exposure:**
- Message appears as Base64-encoded audio
- Audio appears as random noise
- Even if Base64 decoded to WAV, message is LSB-encoded
- Extracting LSB yields binary string (encrypted ciphertext)
- Ciphertext unreadable without session PIN

**Mitigation Layers:**
```
Layer 1: Steganography (obscures message presence)
Layer 2: RSA (distribution of session PIN)
Layer 3: AES-256 (encryption of message content)
Layer 4: HTTPS (prevents unencrypted eavesdropping)
```

**Residual Risk:** Low - Multi-layer protection

#### Scenario 2: Database Breach

**Threat:** Attacker gains MongoDB access, steals all data

**Compromised Data:**
```
users collection:
├─ username ✓ (readable)
├─ email ✓ (readable)
├─ password ✗ (bcrypt hash, unreversible)
├─ public_key ✓ (non-sensitive)
├─ encrypted_private_key ✗ (AES-encrypted with user password)
└─ warningsCount ✓ (metadata only)

sessions collection:
├─ id ✓ (readable but useless)
├─ pin ✗ (labeled "HIDDEN", never stored plaintext)
├─ pin1 ✗ (RSA-encrypted, requires target's private key)
├─ pin2 ✗ (RSA-encrypted, requires target's private key)
└─ status ✓ (metadata only)

offlinemessages collection:
├─ payload ✗ (contains encrypted message + audio data)
└─ created_at ✓ (timestamp)
```

**Attacker's Position:**
- Can enumerate users
- Cannot crack bcrypt passwords
- Cannot decrypt private keys (encrypted with user password)
- Cannot decrypt message content (encrypted PIN unknown)
- Cannot impersonate users (JWT_SECRET on server only)

**Mitigation:** Database encryption at rest (MongoDB Atlas feature)

**Residual Risk:** Very Low - Messages safe even with database compromise

#### Scenario 3: Malicious Admin Intercedes

**Threat:** Server admin with JWT_SECRET tries to decrypt messages

**Attempt 1: Decrypt Private Keys**
- Admin has encrypted_private_key from database
- But it's AES-encrypted with user password
- Admin doesn't have user password
- Result: Cannot decrypt

**Attempt 2: Intercept Session PINs**
- Admin sees encrypted pins (pin1, pin2)
- pin1 is RSA-encrypted with User A's public key
- Admin has User A's public key (stored in database)
- But RSA encryption is asymmetric; can only decrypt with private key
- Admin doesn't have private key (encrypted client-side)
- Result: Cannot decrypt

**Attempt 3: Issue Fake Admin Warning**
- Admin CAN create fake system alerts
- Mitigation: User can contact support to verify alerts' legitimacy

**Mitigation:** 
- Credentials encryption on client
- Don't grant admins user passwords
- Audit logging on sensitive operations

**Residual Risk:** Low - Admin cannot decrypt messages without user password

#### Scenario 4: Malware on User Device

**Threat:** Malware gains access to user's browser memory/localStorage

**Exposure:**
- Malware can read plaintext session PIN from memory
- Malware can read plaintext private key (decrypted during login)
- Malware can intercept messages in memory

**Mitigation:**
- Encourage strong device security
- Store private key encrypted client-side
- Clear sensitive data on logout
- Monitor for suspicious Socket.IO events

**Residual Risk:** High - Client-side compromise is difficult to prevent server-side

**Recommendation:** User should be advised to:
- Use antivirus software
- Keep OS/browser updated
- Avoid untrusted websites while using StegoChat
- Log out when away from device

## Cryptographic Algorithms

### RSA-2048

**Standard:** PKCS #1 v2.1 (RFC 3447)

**Implementation:** Window.crypto.subtle.generateKey (WebCrypto API)

**Key Parameters:**
```javascript
{
  name: 'RSA-OAEP',
  modulusLength: 2048,
  publicExponent: new Uint8Array([1, 0, 1]),
  hash: 'SHA-256'
}
```

**Encryption Scheme:** OAEP (Optimal Asymmetric Encryption Padding)

**Current Status:** ✅ Considered safe until 2030s
- 2048 bits ≈ 112 bits of symmetric strength
- No known effective attacks against RSA-OAEP
- NIST recommends 2048 bits minimum through 2030

**Future Migration Path:**
- 2030+: Consider migration to RSA-4096 or elliptic curve (ECC)
- Current plan: Add ECC support in v2.0

### AES-256-CBC

**Standard:** NIST FIPS 197

**Implementation:** CryptoJS.AES.encrypt

**Key Derivation:** PBKDF2 (Password-Based Key Derivation Function 2)

**Parameters:**
```javascript
{
  keySize: 256,      // bits
  blockSize: 128,    // bits (16 bytes)
  mode: 'CBC',       // Cipher Block Chaining
  padding: 'Pkcs7'   // PKCS#7 padding
}
```

**Current Status:** ✅ Considered safe until 2100s
- 256-bit key = 2^256 possible keys
- No known effective attacks against AES-256
- NIST recommends AES-256 for TOP SECRET classification

### SHA-256

**Standard:** FIPS 180-4

**Usage:** RSA-OAEP hash, JWT signing

**Current Status:** ✅ Still recommended
- 256-bit output
- No known practical attacks
- NIST plans to deprecate around 2100

## Attack Scenarios & Mitigations

### 1. Brute-Force PIN Attack

**Attack:** Attacker intercepts encrypted audio, tries all 100,000 possible PINs

**Analysis:**
```
PIN space: 000000 - 999999 = 1,000,000 possibilities
But encrypted PIN is hardened:
├─ OTP first attempt invalidates on use
├─ Rate limiting: 300 requests/15min globally
├─ Session has max lifetime (implementation dependent)
└─ Network latency: Even at 1000 requests/sec = 1000 seconds
```

**Mitigation:**
- Rate limiting on API and Socket.IO
- PIN invalidation on first use
- Session timeout management

**Residual Risk:** Very Low - 1M attempts at server rate limit takes hours

### 2. Birthday Attack on Session PIN

**Attack:** Attacker tries to find collision in PIN generation

**Analysis:**
- PIN space: 1,000,000
- Attacker with 10,000 sessions: ~1% chance of collision
- Collision would only affect that session pair
- Attacker would need to intercept both audio streams simultaneously

**Mitigation:**
- Use 6-digit PIN (reasonable for user memorability)
- Migrate to 8-digit PIN in v2.0 (100M space)
- Session cleanup on disconnect

**Residual Risk:** Low - Attacker must intercept specific sessions

### 3. Replay Attack

**Attack:** Attacker captures message A→B, replays to B later

**Mitigation:**
- Messages include timestamp (in practice: message order in chat)
- Session-based encryption (replayed message decrypts but appears in wrong order)
- User warning: "Out-of-order message received"

**Residual Risk:** Very Low - User immediately notices duplicate/reordered message

### 4. Cryptanalysis of AES

**Attack:** New mathematical breakthrough cracks AES-256

**Likelihood:** Extremely Low as of 2026
- No known vulnerabilities against AES-256-CBC
- Would require fundamental breakthrough in cryptanalysis
- Quantum computers are decades away

**Mitigation:** Monitor crypto research; quarterly review of algorithm choices

### 5. Social Engineering

**Attack:** Attacker tricks admin into deleting legitimate user

**Scenario:** Attacker claims false identity, requests account deletion

**Mitigation:**
- No delete requests processed without verification
- Audit log all admin actions
- Require admin password confirmation for sensitive actions

**Residual Risk:** Low - Requires compromised admin

## Key Management

### Private Key Generation & Storage

**Generation Location:** Client-side (Browser WebCrypto API)

**Storage Strategy:**
```
Option 1: Encrypted in Browser localStorage
├─ Private key: AES-256 encrypted with user password
├─ Encryption: CryptoJS.AES.encrypt(privateKey, userPassword)
├─ Decryption: CryptoJS.AES.decrypt(encrypted, userPassword) during login
├─ Pro: Persistent across sessions
└─ Con: Vulnerable if device compromised

Option 2: Memory-only (Current Implementation)
├─ Private key: Loaded on login, stored in memory
├─ Cleared on logout
├─ Pro: Safer from device compromise
└─ Con: Lost if page refreshes unexpectedly

Recommended: Hybrid (v2.0)
├─ Store encrypted key in localStorage
├─ Keep decrypted copy in memory during session
├─ Auto-logout on browser inactivity
```

### Key Rotation Strategy

**Current:** ⚠️ No automatic key rotation
**Recommended for v2.0:**
```
Annual Key Rotation:
├─ Admin initiates rotation
├─ Users prompted to log out
├─ Generate new RSA-2048 key pair
├─ Store encrypted private key
├─ Update public key on server
└─ Old key archived for message decryption (30-day grace period)
```

### Key Backup & Recovery

**Current:** ⚠️ No backup mechanism
**Recommendation:**
- Users should export RSA private key (encrypted) to secure location
- Add "Download Key Backup" feature in settings
- Implement "Recover from Backup" mechanism

## Security Best Practices

### For Developers

1. **Never Log Sensitive Data**
   - Don't log private keys, passwords, or plaintext messages
   - Review server logs weekly for violations

2. **Dependency Management**
   - Run `npm audit` before each deployment
   - Update dependencies monthly
   - Test updates in staging before production

3. **Code Review**
   - Peer review all cryptographic code
   - Security review before production deployment
   - Third-party security audit recommended

4. **Secrets Management**
   - Use environment variables for sensitive config
   - Never commit `.env` files to Git
   - Rotate JWT_SECRET every 90 days

5. **Error Handling**
   - Don't expose stack traces to clients
   - Log errors server-side
   - Return generic error messages to users

### For Users

1. **Password Security**
   - Use unique, strong password (12+ chars)
   - Don't share password with anyone
   - Change password if suspected compromise

2. **Device Security**
   - Keep OS and browser updated
   - Use antivirus software
   - Don't access from untrusted devices

3. **Account Security**
   - Log out after use (especially shared devices)
   - Review active sessions
   - Report suspicious activity to admin

4. **Communication Security**
   - Warn contacts about suspicious behavior
   - Verify identity through out-of-band channel before sensitive discussions
   - Don't click suspicious links in messages
